# 3. 单词折行边界：光标紧贴词尾、不画圆点

日期：2026-06-20
状态：已接受

## 背景

单词模块的练习文本是一整行（无换行符），由 `wrapGhostWordBlockLoose` 按终端宽度折成多视觉行；空格被渲染成可见圆点 `·`（`space_glyph`）。

折行处，每个视觉行末尾那个词的"待打分隔空格"曾经被原样保留（圆点 `·` + 光标态）。结果：用户打完该行最后一个词后，光标显示在词后的圆点 `·` 上——看起来像"越过词、停在一个圆点上、卡在本行没换行"。用户明确反馈这让他误以为空格没生效，从而退格重打（见 QA #1）。

但简单地"行末不渲染待打空格"会让光标在折行处**不可见**——这恰好是历史上修过的一个 bug，并有测试守护（"keeps the cursor visible on a wrapped repeated long-word separator"）。所以不能直接删。

候选方案：A) 光标挪到下一行行首；B) 折行处光标消失；C) 维持现状（停在圆点）；D) 光标紧贴词尾、行末空格不画圆点。

## 决策

采用 **D**：折行视觉行末尾的待打分隔空格**保留**（从而光标可见），但把空格圆点 `·` 还原为普通空格——光标成为紧贴词尾的高亮块，不再像越过词停在圆点上。完全未打到的行末空格（全 `pending`）仍不渲染，保持行尾干净。

实现集中在 `looseSeparatorCellsForRowItem`：行末项的分隔单元，`every(pending)` → 丢弃；否则保留但将 `·` 映射为 `" "`。

## 后果

- 光标紧贴词尾、敲空格后跳到下一行的下一个词，不再卡在误导性的行末圆点上，也不会消失。
- 行内（非行末）的空格仍显示圆点 `·`，不受影响。
- 未开 `space_glyph` 的场景（如旧测试 684）分隔本就是普通空格，行为不变。
- `looseSeparatorCellsForRowItem` 的行末特殊处理是**刻意**的，勿当作冗余"优化"掉——本 ADR 即理由。

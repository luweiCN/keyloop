# 自适应综合训练（诊断-处方引擎）设计

日期：2026-06-13
状态：已与用户确认设计，待实现

## 1. 背景与问题

现有综合训练存在四个问题：

1. **算法黑盒**：一级菜单"综合练习"与各模块 mix 的选取逻辑分散在 `targets.ts`（2000+ 行）各处，无法向用户解释"为什么今天练这些"。
2. **覆盖缺口**：一级综合训练不包含日常-文章（`everyday_articles`）、日常-单词拆解（`everyday_word_decomposition`）、自建语料库；后续新增科目也不会自动进入综合训练。
3. **时长与速度脱节**：每日目标硬编码 20 分钟（`targets.ts:587`），语料量是写死的常数，不随用户在各模块的实际打字速度变化。打字快的人 10 分钟练完，慢的人要 40 分钟。
4. **弱项串味**：错误内容跨形态污染——练单词时打错的单词会被塞进符号等不相干的训练组。

## 2. 核心理念：诊断 → 处方

> 不是优化综合训练的选题算法，而是根据用户的打字数据做**诊断**，并针对诊断生成**针对性的治疗**。

- **诊断对象是底层技能，不是训练科目**。科目只是药。一个只练代码的用户，从其代码会话的字符级数据（`key_events`、`error_chars`、`token_stats`）同样能诊断出"分号慢、数字错误率高、大写切换卡顿"。技能信号跨模块通用。
- **随机均衡打底，弱项加权抬升**：基础份额保证所有训练形态轮换练到；技能弱项决定哪些阶段这段时间权重抬高；技能恢复正常后权重自动落回。

## 3. 诊断层（新文件 `src/training/diagnosis.ts`）

纯函数：输入全部历史会话（不分模块），输出结构化《技能画像》。不做任何"该练什么"的决定。

### 3.1 技能维度

| 技能维度 | 数据来源 | 衡量什么 |
|----------|----------|----------|
| 字母键区（中排/上排/下排、左手/右手） | `key_events` 按键归区 | 各区击键延迟、错误率 |
| 数字 | `key_events` | 数字键速度与准确度 |
| 符号标点 | `key_events` + `error_chars` | `;{}()=>` 等键的表现 |
| 大小写切换 | shift 组合键事件 | 驼峰、句首大写的卡顿 |
| 常见词流畅度 | `token_stats` | 普通英文单词的连贯速度 |
| 长词/生词 | `slow_tokens`、长 token | 长单词是否打得很慢 |

样本是**字符/词级事件**而非会话，因此用户只练一个模块时所有技能维度也都有数据。

### 3.2 指标计算

- **窗口**：按"最近 N 次"取窗口（每技能维度最近 10 次相关会话的事件），不按天数。高频用户窗口自然短，低频用户保证样本量。
- **EWMA**：指数加权平均，半衰期 4 次（`w_i = 0.5^(i/4)`）。速度与准确率同窗口同权重。
- **趋势**：窗口前半 vs 后半对比，变化超过 ±8% 才判 `improving` / `declining`，否则 `stable`；样本 <4 次为 `insufficient`。
- **状态**：
  - `weak`：EWMA 准确率 <92% 或错误率 ≥8%，或趋势 declining
  - `stable`：样本 ≥3、准确率 ≥97%、错误率 ≤2.5% 且趋势非 declining
  - `unrated`：样本不足（不算弱项，不触发加时）
  - 其余 `normal`

### 3.3 形态速度（供字符预算换算）

按训练形态（键位/单词/符号/句子/文章/代码）分别统计该形态会话的 EWMA WPM（同样最近 10 次、半衰期 4）。速度计算一律基于 `active_ms`（排除发呆时间），沿用 `correct_chars / 5 / minutes`。

### 3.4 Focus 池按形态分桶（修复串味）

诊断报告中的内容回流池按训练形态分桶：

- `focus_words`（仅供单词形态）
- `focus_sentences`（仅供句子形态）
- `focus_code`（仅供代码形态）
- `focus_chars`（技能特征：键位/符号/数字/大小写，全阶段可用于调整语料**特征含量**）

**硬规则："技能跨阶段，内容不跨形态"**：

- 技能信号（字符级）可以跨阶段调整语料特征——大小写弱 → 单词阶段多选驼峰词；
- 内容信号（具体的词/句/代码片段）只允许回流到同形态阶段——错的单词进之后的单词阶段，禁止出现在符号等其他阶段。处方层只读同桶数据，从类型上杜绝串味。

### 3.5 其他输出

- 近 7 天实际日均活跃练习分钟（中位数），供推荐时长。
- 跨层对比诊断：字符级技能都正常但某模块整体 WPM 明显偏低 → 提示"问题在阅读/上下文而非手指"，处方多排该模块。

## 4. 处方层（新文件 `src/training/prescription.ts`）

纯函数：输入技能画像 + 用户偏好，输出当日《阶段计划》。

### 4.1 科目启用

`preferences.json` 新增 `enabled_modules`：按一级模块开关（基础输入/日常英语/编程基础/代码实战），默认全开。非程序员可关闭编程类，处方永不开出代码阶段。技能弱项的治疗候选按优先级排列、跳过禁用项（如符号弱 → 编程:符号数字被禁则用基础输入:符号行），保证弱项总有药可开。

### 4.2 今日目标时长

```
今日目标 = base + 弱项数 × 5 分钟
  base：技能画像全面稳定（高手维持模式）→ 10 分钟
        其他（普通/新用户）→ 15 分钟
习惯柔性上限：≤ max(15, 近 7 天日均练习 × 1.5)
clamp 到 [10, 45] 分钟
```

- 下限 10 分钟 = 高手维持手感；上限 45 分钟 = 弱项多且本身有大练习量习惯的进步期用户。
- 高手练得短：时长由"还有多少要治"驱动，不由习惯驱动；习惯只做上限约束（不给每天只练 8 分钟的人推荐 40 分钟）。
- 今日目标只是**进度线**，不是上限。练完一组可再进一组，超额随意。

### 4.3 阶段序列（按训练形态，非科目）

一次综合训练 = 一组动态 4~6 个阶段：

| # | 阶段 | 语料来源（启用科目的语料池合并） | 出现规则 |
|---|------|--------------------------------|----------|
| ① | 键位指法热身 | foundation 各键位科目；弱键区决定练哪排/哪手 | 必有，短 |
| ② | 单词 | 日常单词 + 编程术语 + 技术长词 + 自建词库；拆解轮换日并入此阶段 | 必有 |
| ③ | 符号与数字 | 数字行 + 符号行 + 编程符号数字 | 弱项时必有，否则轮换 |
| ④ | 句子 | 日常句子 + 自建库句子 | 常有 |
| ⑤ | 文章 | 日常文章 | 约 3 天轮换一次 |
| ⑥ | 代码 | 代码块/函数/文件片段 | 启用编程类科目才有 |

新增科目只需归类到某个形态的语料池，自动进入综合训练——结构性消除覆盖缺口。

### 4.4 阶段时长权重

```
每个出现的阶段基础权重 1.0
被技能弱项命中 → ×1.5
对应技能已 stable 且无弱项命中 → ×0.5（降频不消失）
归一化后乘今日目标时长
```

每个阶段附带**理由**字符串（如"数字行错误率 9%"/"常规轮换维持手感"），供 UI 展示。

### 4.5 字符预算换算

```
阶段字符预算 = 阶段分钟数 × 该形态 EWMA WPM × 5
冷启动 fallback WPM：键位 18 / 单词 22 / 符号 16 / 句子 22 / 文章 22 / 代码 14，再 ×0.8（故意低估）
```

各 mix 生成器改造为**形态生成器**：接收（字符预算、特征侧重、同形态 focus 池），按预算缩放内容量，替换现有写死常数。

### 4.6 长内容轮换

从 `daily_runs.json` 历史计划反查上次出现日期：文章间隔 ≥3 天则排入、单词拆解间隔 ≥2 天则并入单词阶段。不引入新状态文件。间隔为初始经验值，可调。

## 5. 时间控制：三层防线 + 一条原则

**原则：时间是预估，内容才是契约。** 阶段一旦开始内容固定，打完算完，不做到点强切（保护心流；预估错是系统的问题，不转嫁为用户挫败感）。

1. **事前**：预估与统计一律用 `active_ms`（发呆不污染 WPM）；冷启动预算 ×0.8 保守低估——"练完意犹未尽"优于"练不完劝退"，两种错误代价不对称。
2. **事中（懒生成红利）**：每阶段语料进入时才生成。阶段 N 结束后用本会话实测速度重算后续阶段预算，并向今日目标对齐（必要时缩量或砍掉"常规轮换"阶段，保留"弱项加强"阶段）。速度被严重低估时同一会话内即追平。
3. **事后**：每阶段产生一条会话记录进入形态 EWMA，半衰期 4 次 → 2~3 次练习后估值贴住真实水平，预估误差自耗散。

进度线（"今日 18/24 分钟"）一律按实际活跃时间计，永远真实。

## 6. 交互与 UI

### 6.1 诊断/计划屏（进入综合训练后、开打前）

```
┌ 今日训练计划 ──────────────────────────────┐
│ 诊断摘要：                                  │
│   符号键错误率 8.4% ↑（弱项）               │
│   数字行速度 12 WPM，低于整体水平           │
│   单词流畅度良好，大小写切换正常            │
│                                            │
│ 今日计划：5 个阶段，约 24 分钟              │
│   ① 键位热身    2 分钟  （常规热身）        │
│   ② 单词        6 分钟  （复现昨日错词 ×4） │
│   ③ 符号与数字  7 分钟  （弱项加强）        │
│   ④ 句子        5 分钟  （常规轮换）        │
│   ⑤ 代码        4 分钟  （维持手感）        │
│                                            │
│ [Enter] 开始   [←→] 调整时长   [Esc] 返回   │
└────────────────────────────────────────────┘
```

- 每个阶段展示选它的理由——黑盒问题的最终解。
- 推荐时长可用方向键当场调整，改后语料量按新时长重新换算。

### 6.2 阶段间歇屏

本阶段 WPM/准确率 + "阶段 2/5 完成，今日 8/24 分钟。[Enter] 下一阶段 [Esc] 退出（进度已保存）"。

### 6.3 进度保存与恢复

- 中途退出：保存阶段计划骨架 + 各阶段完成状态（扩展 `daily_runs.json` 的 plan 结构 + 现有 `current_session.json` checkpoint），再进入从未完成阶段接着练，已完成阶段不重打。
- 阶段内中断沿用现有 checkpoint 恢复机制。

## 7. 数据流落点

- 新增 `src/training/diagnosis.ts`（技能画像）、`src/training/prescription.ts`（阶段计划），均为纯函数，从 `targets.ts` 拆出规划逻辑。
- `daily_runs.json`：plan 结构扩展为阶段计划 + 完成状态。
- `preferences.json`：新增 `enabled_modules`。
- 各 mix 生成器 → 形态生成器（字符预算 + 特征侧重 + 同形态 focus 池参数化）。
- 一级菜单"综合练习"入口（`appModel.ts:696`）改走 诊断 → 处方 → 诊断屏 → 逐阶段懒生成 流程。
- 各二级菜单的 mix 项保留为"单形态快捷练习"，复用对应形态生成器（同样吃诊断的 focus 池与特征侧重）。

## 8. 与现状的行为变更清单

1. 稳定模块从"直接跳过"改为"权重 ×0.5 降频出现"。
2. 20 分钟硬编码 → 动态推荐时长 [10, 45]，可当场调整。
3. 语料量从写死常数 → 按形态 EWMA WPM × 时长换算。
4. 文章、单词拆解、自建语料库纳入综合训练（轮换/合并进对应形态阶段）。
5. focus 池从全局混用 → 按形态分桶，内容不跨形态回流。
6. `buildDailyPracticePlan().lessons[0]` 只取第一课的旧行为，由"逐阶段懒生成"取代。
7. 性能窗口从 21 天 → 最近 10 次 EWMA（半衰期 4 次）。

## 9. 测试策略

诊断与处方均为纯函数，构造历史会话数据直接断言：

- 给定符号错误率 9% 的历史 → 符号阶段必须出现且加权。
- 错词 "algorithm" 只允许出现在单词阶段的语料中。
- 禁用编程类科目 → 计划中永远没有代码阶段，符号弱项回退到基础输入:符号行。
- 无历史数据 → 15 分钟默认计划，全维度 unrated，冷启动预算 ×0.8。
- 全面稳定的历史 → 10 分钟维持计划，稳定阶段权重 ×0.5。
- 文章上次出现 2 天前 → 今日不排文章；3 天前 → 排入。
- 阶段实测速度偏离预估 → 后续阶段预算按实测重算且总时长向目标对齐。
- 时长边界：习惯 8 分钟/弱项 3 个 → clamp 与习惯上限生效。
